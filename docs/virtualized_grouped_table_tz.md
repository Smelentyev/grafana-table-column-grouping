# ТЗ: Большая переработка grouped-таблицы с виртуализацией `tbody`

## Контекст

Текущая реализация grouped-режима использует HTML-таблицу для поддержки сложных многоуровневых заголовков с `rowSpan` и `colSpan`. Это архитектурно оправдано, но при больших объёмах данных производительность ограничивается:

- большим числом одновременно отрисованных DOM-узлов;
- полной отрисовкой `tbody` для текущей страницы;
- дополнительными затратами на сортировку, фильтрацию и форматирование значений;
- сложностью дальнейшего масштабирования grouped-режима.

На текущем этапе уже реализованы:

- исправления раскладки коротких и глубоких групп;
- оптимизация data pipeline через индексы строк;
- настройки явной пагинации;
- настройка количества строк на страницу;
- тестовый repro-сценарий на 1000 строк.

Следующий шаг: сделать крупный рефакторинг grouped-режима так, чтобы таблица оставалась гибкой по layout, но стала масштабируемой на больших датасетах.

## Цель

Разработать новую архитектуру grouped-таблицы, в которой:

- сложный `thead` с произвольной группировкой сохраняется;
- `tbody` поддерживает виртуализацию;
- производительность на больших объёмах данных заметно улучшается;
- существующие возможности grouped-режима не теряются.

## Основная идея

Сохранить нативный HTML-`table` для заголовка, но переработать тело таблицы:

- вынести вычисление структуры данных в отдельный pipeline;
- нормализовать модель колонок, заголовков и строк;
- рендерить только видимое окно записей в `tbody`;
- синхронизировать виртуализированное тело с ширинами колонок и sticky header.

## Область работ

### 1. Архитектура данных

Нужно выделить отдельный слой подготовки данных grouped-таблицы:

- модель заголовков;
- модель колонок;
- модель визуальной строки;
- план ячеек для `tbody`;
- pipeline фильтрации, сортировки, пагинации и виртуализации.

Результат: JSX-рендер должен работать в основном с уже подготовленными структурами, а не вычислять логику на лету.

### 2. Виртуализация `tbody`

Нужно реализовать windowing для grouped-режима:

- рендер только видимых записей;
- настраиваемый `overscan`;
- верхний и нижний spacer;
- корректная работа при `maxFieldDepth > 1`.

Под “одной записью” понимается логическая строка данных, которая может занимать несколько визуальных строк в `tbody`.

### 3. Сохранение layout заголовка

Нужно сохранить:

- текущую модель `root-group` / `group-container` / `column`;
- поддержку `vertical` и `horizontal`;
- корректный `rowSpan`/`colSpan`;
- sticky header;
- ресайз колонок.

### 4. Синхронизация ширин

Нужно сохранить и усилить текущую модель ширин через `<colgroup>`:

- `thead` и `tbody` должны использовать единый источник истины для ширин;
- ресайз должен сразу отражаться на виртуализированном `tbody`;
- нельзя допустить визуального расхождения групповых заголовков и ячеек данных.

### 5. Фильтрация и сортировка

Нужно адаптировать текущие оптимизации под новую архитектуру:

- фильтрация должна работать по индексам строк;
- сортировка должна минимизировать промежуточные аллокации;
- popup-фильтры желательно перевести на lazy + cached вычисление значений;
- итоговая виртуализация должна работать поверх уже отфильтрованных и отсортированных индексов.

## Нефункциональные требования

### Производительность

Целевые ориентиры:

- 1000 строк в grouped-режиме должны работать заметно плавнее текущей реализации;
- 5000 строк не должны приводить к взрывному росту DOM;
- открытие фильтров, сортировка и ресайз должны оставаться отзывчивыми;
- при прокрутке не должно быть заметных скачков layout.

### Совместимость

Нельзя ломать:

- текущие настройки панели;
- существующие структуры `columnGrouping`;
- сценарии без группировки;
- работу field config и display processors.

### Тестируемость

Нужно увеличить покрытие тестами:

- unit-тесты для pipeline;
- тесты на план строк и ячеек;
- тесты на виртуальное окно;
- regression-тесты на короткие и глубокие группы;
- e2e / screenshot smoke-test на grouped-режим с большим набором данных.

## Предлагаемая архитектура

### Новый pipeline grouped-таблицы

Рекомендуется вынести в отдельные модули:

- `headerPlan` — построение структуры заголовков;
- `columnPlan` — нормализованный список колонок;
- `rowPlan` — визуальная модель логической записи;
- `cellPlan` — описание ячеек для рендера `tbody`;
- `virtualWindow` — диапазон видимых записей.

### Новый рендер `tbody`

Рекомендуется:

- не итерироваться по всем строкам страницы;
- не собирать JSX для невидимых записей;
- держать минимальный объём активного DOM;
- отдельно рендерить spacer-элементы для компенсации полной высоты.

## Этапы реализации

### Этап 1. Подготовка архитектуры

- распилить `TableWithGroupedHeaders.tsx` на pipeline и view;
- вынести модели и helper-типы;
- изолировать header/body rendering.

### Этап 2. Виртуализация тела

- реализовать вычисление окна видимых записей;
- подключить overscan;
- добавить spacer-механику;
- синхронизировать scroll-container.

### Этап 3. Оптимизация вычислений

- кэш display values;
- lazy/cached filter options;
- минимизация `Field` lookup в рендере;
- сокращение React-логики в цикле body render.

### Этап 4. Тестирование и стабилизация

- unit-тесты pipeline;
- regression-тесты для grouped-layout;
- perf smoke-test на 1000 строк;
- ручная проверка sticky header, resize, pagination, filter popup.

## Критерии приёмки

Задача считается выполненной, если:

- grouped-режим продолжает корректно отображать многоуровневые заголовки;
- `tbody` рендерит только видимое окно записей;
- на большом наборе данных DOM не растёт линейно от общего числа строк;
- функциональность сортировки, фильтрации, пагинации и ресайза не ломается;
- unit-тесты и typecheck проходят;
- repro-сценарий на 1000 строк отображается корректно и работает быстрее текущей реализации.

## Вне рамок этой задачи

Не включать в первый этап:

- полный отказ от HTML-таблицы в пользу div-grid;
- полную замену grouped-режима на стороннюю библиотеку;
- сложные анимации;
- отдельный print/export renderer.

## Риски

- сложность синхронизации `thead` и виртуализированного `tbody`;
- нестабильность при переменной высоте строк;
- вероятность регрессий в сложных структурах группировки;
- рост сложности кода без достаточного покрытия тестами.

## Результат разработки

Ожидаемый итог:

- новая версия grouped-таблицы с виртуализированным `tbody`;
- сохранение текущего гибкого layout заголовков;
- заметное улучшение производительности на больших данных;
- устойчивый pipeline, который проще развивать дальше.
